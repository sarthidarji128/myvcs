Local-only version control

A ledger for your code that never asks for a signal.

myvcs is a minimal version control system that lives entirely on disk. No remotes, no daemons, no network calls — just a command-line tool for snapshotting, tracking, and reverting a project from an air-gapped machine, a flaky connection, or a folder that was never meant to be pushed anywhere.

The problem

Version control assumed you'd always be online.

Most tools in daily use — including the one this project deliberately avoids depending on — were built around a client/server model: cloning, pushing, pulling, and remote tracking all expect a network on the other end. myvcs strips that assumption out and keeps every operation local to the folder you're working in.

Typical networked workflow

  • → Cloning a remote needs a live connection at least once
  • → Pushing/pulling assumes a reachable server
  • → Remote-tracking branches add configuration overhead
  • → Air-gapped or offline machines need workarounds (bundles, patches)

myvcs

  • ✓ setup registers a project directory — nothing leaves the disk
  • ✓ Every command reads and writes to .myvcs/ only
  • ✓ No remotes, no auth, no server process to keep alive
  • ✓ Works identically on a plane, a lab machine, or a locked-down network
How it's built

A working directory, hashed objects, and one JSON ledger.

myvcs splits its job across two files: myvcs.py, the engine that hashes content and writes commits, and myvcs.sh, the CLI wrapper you actually type commands into.

1

Registration, not scanning. setup <name> <path> writes an entry to .myvcs_config.json so myvcs always knows which folder a project name refers to — you switch between projects with use instead of cd-ing and re-initializing.

2

Content-addressed storage. Files are hashed with SHA-1 before they're written to objects/, so two commits that share an unchanged file don't duplicate its bytes on disk.

3

One readable ledger. commits.json holds the entire history as plain JSON — no packfiles, no binary index, easy to inspect or back up by hand.

4

Status and diff read from that ledger. Comparing the working tree against the last commit is just a line-by-line comparison against the stored snapshot — no remote round-trip is ever possible, because there's nowhere else to ask.

      Reference

      The seven commands that make up the CLI.

      Register a project

      Points myvcs at a directory and gives it a short name so every future command can refer to it without a full path.

      • Writes an entry into .myvcs_config.json
      • Does not modify the target directory until you run a command inside it
      • Safe to run again later to register additional projects
      terminal
      $ ./myvcs.sh setup myproject /path/to/your/project
✓ registered "myproject" → /path/to/your/project

      Switch the active project

      Every following command operates on whichever project was last selected with use — no need to cd anywhere first.

      • Reads the registry written by setup
      • list shows every registered project
      terminal
      $ ./myvcs.sh use myproject
✓ active project set to "myproject"

      See what changed

      Compares every tracked file's current content hash against the version stored in the last commit.

      • Flags modified, new (untracked), and deleted files separately
      • Nothing is written to disk — status is read-only
      terminal
      $ ./myvcs.sh status
modified:  notes.md
new:       scratch.py

      Line-by-line comparison

      Runs a line-based diff between the working copy of a file and the version captured in the last commit.

      • Added lines are marked distinctly from removed ones
      • Unchanged context lines are shown around each change
      terminal
      $ ./myvcs.sh diff notes.md
- Draft notes for v0.3
+ Draft notes for v0.4
+ - fixed revert edge case

      Save a snapshot

      Hashes every tracked file, stores any new content in objects/, and appends an entry to commits.json.

      • Requires a message describing the change
      • Each commit records a parent, so history stays a chain
      terminal
      $ ./myvcs.sh commit "Describe changes"
✓ c4a91f2 committed — Describe changes

      Walk the history

      Lists every commit newest-first: its short hash, message, and timestamp.

      • Reads directly from commits.json
      • No pagination or network fetch — the whole ledger lives on disk already
      terminal
      $ ./myvcs.sh log
c4a91f2  Describe changes
a91c2f0  Initial commit

      Roll back

      Restores every tracked file to exactly the content stored in a chosen prior commit.

      • Accepts a full or short commit reference
      • Overwrites the working directory — commit first if you want to keep the current state
      terminal
      $ ./myvcs.sh revert c1
✓ working directory restored to commit c1
      Get running

      From clone to first commit in six lines.

      bash
      $ git clone https://github.com/sarthidarji128/myvcs.git
$ cd myvcs
$ chmod +x myvcs.sh
$ ./myvcs.sh setup myproject /path/to/your/project
$ ./myvcs.sh use myproject
$ ./myvcs.sh status
$ ./myvcs.sh commit "Describe changes"
$ ./myvcs.sh log

      Before you ask

      A few likely questions.

      Not for collaborative or remote-hosted work — Git's ecosystem of hosting, pull requests, and distributed sync is out of scope here. myvcs targets the narrower case of tracking a single local project with zero network dependency.

      Uncommitted changes are overwritten by the target commit's snapshot. Running status before revert is the safest habit.

      Each file's content is hashed with SHA-1 before it's written to objects/. If a commit doesn't change a file, its existing object is reused instead of being written again.

      Yes — setup can register as many named projects as you like, and use switches the active one at any time.
